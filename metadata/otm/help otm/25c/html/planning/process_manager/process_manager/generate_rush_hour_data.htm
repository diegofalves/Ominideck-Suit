Generate Rush Hour Data

The Generate Rush Hour Data process is accessed via Contract and Rate Management > Process Management > Generate Rush Hour Data.

TheÂ Generate Rush Hour Data process allows you to use historical traffic data in creating shipments. This process generates data in the RUSH_HOUR table for a specified geography and time period, by calling a third party external service engine, such as PC*MILER (ALK) or HERE. The rush hour delays for a specific lane at a specific time are calculated by calling the external service engine without traffic effects and then calling it again with traffic effects for the specific time.

Note: If using PC*Miler to generate rush hour data for European locations, you must use version 30 (or newer) This is defined by setting the propertyÂ glog.ExternalDistanceEngine.PCMilerWS.DataVersion to 30 (or higher).

Note: If using HERE to generate rush hour data, there are two options

The process creates rush hour data for 15 minute time periods. This is not configurable. OTM provides public Rush Hour Calendar data, specifically for Rush Hour Calendars. Each Rush Hour calendar specifies a single 15 minute period for a specific day-of-week and time-of-day, using the activity type RUSHHOURS. Each new RUSH HOUR entry uses one of these public Rush Hour calendars.

OTM ships with 672 public rush hour calendars. (4 (increments per hour) * 24 * 7 = 672 Calendars.). Public calendars are named according to the syntax "RUSH HOUR <DAY> <HOUR><MINUTE>". For example: "RUSH HOUR MONDAY 0800", "RUSH HOUR MONDAY 0815". Note that these calendars and their names should not be changed.

The input Calendar ID specifies the time periods for which rush hour data is generated. It is not the calendar entry for the new RUSH HOUR data. This input calendar should be configured with the activity type RUSHHOURS (perform). The input calendar, like any calendar, can specify different periods on different days. Typically it is configured to specify rush hour periods for a given day (1 day period), or for a given day-of-week (7 day period).

The "override" dates on the input calendar are ignored since OTM is not creating PUBLIC Rush Hour Calendars for override dates.

The start date on the input calendar defines the week for which the new RUSH HOUR data is computed. Choose a week that has no local holidays for the lanes in question, as the external service engine (ESE) includes historical data for holidays that you do not want to capture. (For example, you will not use the week including July 4 in the US, because the process will then be using the July 4 holiday rush hour data for that particular day-of-week, e.g., Tuesday, which would be wrong.)

The External Service Engines require a source location and a destination location in order to provide transit time data. HERE External Service Engine has two services that can be used to query transit times between locations:

  • Calculate Route: This service takes in a location pair as an input, and returns the travel time between the source and destination. When the property glog.ExternalDistanceEngine.HERE.authenticationType is set to ApiKey and HERE_LATLONG_TO_LATLONG ESE is selected, the default service URL is fetched from the here.route.host property. When the property glog.ExternalDistanceEngine.HERE.authenticationType is set to OAuth, and if the HERE_LATLONG_TO_LATLONG or HERE_ROUTING_LATLONG_TO_LATLONG ESE is selected, the URL is fetched from the external system configured in the ESE.

  • Calculate Matrix: Gets the transit time between locations and generates rush hour data from those times. The benefit of using this service is the fact that the system can supply a list of source locations, and a list of destination locations, and make one query to the HERE engine to get travel times.Â To use this service, first change the glog.business.session.useMatrixServiceToGenerateRushHours property to true. When the property glog.ExternalDistanceEngine.HERE.authenticationType is set to ApiKey, and HERE_LATLONG_TO_LATLONG ESE is selected, the default service URL is fetched from the here.route.matrix.host property. When the property glog.ExternalDistanceEngine.HERE.authenticationType is set to OAuth, and if the HERE_LATLONG_TO_LATLONG or HERE_ROUTING_MATRIX_LATLONG_TO_LATLONG ESE is selected, the URL is fetched from the external system configured in the ESE. This new service might not always improve performance.

    WhenÂ glog.business.HereRoutingBatch.isAsyncMatrixCall is set to true, the HERE V8 Matrix Asynchronous API is used while calculating the travel times using the matrix.Â 

Note: The ESE, HERE_LATLONG_TO_LATLONG is for the V7 version of HERE API whereas HERE_ROUTING_LATLONG_TO_LATLONG, andÂ HERE_ROUTING_MATRIX_LATLONG_TO_LATLONG are for the V8 version of HERE API. To use the matrix service in HERE V8, you should select theÂ HERE_ROUTING_MATRIX_LATLONG_TO_LATLONG ESE.

Regions and Lanes

The Generate Rush Hour Data process provides several inputs. The input Source/Destination Region Group ID, Source/Destination Location Profile ID, and Consider Lanes Bi-directional checkbox indicate the relevant geographies for the Generate Rush Hour Data process. The generated RUSH HOUR data needs to have lanes, and these lanes may need to be generated by the process. Locations need a latitude and longitude to be valid.

A Region Group is a set of regions. The generated RUSH HOUR data will have lanes based on the regions from the group. These lanes will be:

  • Inter-Region / Location Lanes: These are lanes from each region (or location profile location) to each other region (or location profile location) (but not from a region to itself).
  • Intra-Region Lanes: These are lanes from each region to itself. (That is, each lane is for driving from a location in a region to another location in the same region.)

From the input Source Region Group ID, Destination Region Group ID, Source Location Profile ID, and Destination Location Profile ID the list of lanes for this run of the Generate Rush Hour Data process are determined. For example:

  • Only Source Region Group (example regions, SR1 and SR2): includes inter-regional and intra-regional lanes within the source region group. Will always consider bi-directional lanes regardless of what the Consider Lanes Bi-directional checkbox is set to. For example,
    • SR1 to SR2
    • SR2 to SR1
    • SR1 to SR1
    • SR2 to SR2
  • Only Source Location Profile (SL1 and SL2): includes lanes for each location in the source location profile to each other location in the source location profile. Will always consider bi-directional lanes regardless of what the Consider Lanes Bi-directional checkbox is set to. For example,
    • SL1 to SL2
    • SL2 to SL1
  • Source Location Profile (SL1 and SL2) and Source Region Group (SR1 and SR2): includes inter-regional and intra-regional lanes within the source region group, lanes for each location in the source location profile to each other location in the source location profile, and from every region in the source region group to every location on the source location profile. This applies to the reverse direction as well. Will always consider bi-directional lanes regardless of what the Consider Lanes Bi-directional checkbox is set to. For example,
    • SR1 to SR2
    • SR2 to SR1
    • SR1 to SR1
    • SR2 to SR2
    • SL1 to SL2
    • SL2 to SL1
    • SR1 to SL1
    • SR2 to SL1
    • SR1 to SL2
    • SR2 to SL2
    • SL1 to SR1
    • SL1 to SR2
    • SL2 to SR1
    • SL2 to SR2
  • Source Region Group (SR1 and SR2) and Destination Region Group (DR1 and DR2): includes lanes from every region in the source region group to every region on the destination region group, as well as intra-region lanes. For example, if Consider Lanes Bi-directional is not selected the following lanes are used:
    • SR1 to DR1
    • SR1 to DR2
    • SR2 to DR2
    • SR2 to DR1
    • SR1 to SR1
    • SR2 to SR2
    • DR1 to DR1
    • DR2 to DR2
  • Source Region Group (SR1 and SR2) and Destination Location Profile (DL1 and DL2): includes lanes from every region in the source region group to every location on the destination location profile, as well as intra-region lanes. For example, if Consider Lanes Bi-directional is not selected the following lanes are used:
    • SR1 to DL1
    • SR1 to DL2
    • SR2 to DL1
    • SR2 to DL2
    • SR1 to SR1
    • SR2 to SR2
  • Source Region Group (SR1 and SR2), Source Location Profile (SL1 and SL2), Destination Region Group (DR1 and DR2), and Destination Location Profile (DL1 and DL2) which includes the following lanes if Consider Lanes Bi-directional is not selected.

Inter-Region Lanes

When calling the external service engine for an inter-region lane, the call uses the Representative Locations for the regions. If a region has no representative location, it is not used to generate any lanes (or identify any lanes that can be re-used). The system also uses every location in the input Location Profile. (Note: OTM only allows Location Profiles with the "Compatible" checkbox selected. This is to avoid a very easy error whereby almost every location in the domain is accidentally brought into the process.)

Scenario

Each time the new Generate Rush Hour Data process is run, OTM generates new RUSH HOUR data. However, you may be running the process several times in order to test or to determine the correct inputs, or because the External Service Engine data has been updated.

The scenario ID is an identifier used to identify the set of Rush Hour data generated for a particular scenario. The SCENARIO_NAME is not a GID, but a user-defined string. The Scenario ID matches the SCENARIO_NAME column for the RUSH_HOUR table.

The first step of the Generate Rush Hour Data process is to delete any rush hour data with the same scenario name as the input Scenario ID.

So, for example, you have a scenario ID of HOUSTON. As you run and re-run the Generate Rush Hour Data process with this scenario, the old rush hour data for the previous run is replaced with the newly generated rush hour data. (This is regardless of whether the other inputs have changed or not.)

If some primary keys of the generated Rush Hour Data match some of the existing entries in the RUSH_HOUR table, OTMÂ updates that entry. This implies that scenario_name, rush_hour_transit_time_delay etc. can change for an existing entry in the RUSH_HOUR table. For example:

Existing Entry: X_LANE_GID = ABC, IS_APPLY_TO_SRC=Y, CALENDAR_GID= RUSH HOUR MONDAY 0600, RUSH_HOUR_TRANSIT_TIME_DELAY = 400, SCENARIO_NAME = "TEST"

Now run this process with the Scenario ID of "TEST2". OTM generates data with the same primary key as the above entry, SCENARIO_NAME is updated to "TEST2".

It is useful to think of different scenarios as different geographies.

Generating Rush Hour Data

  1. In the External Service Engine ID field, specify which External Service Engine (ESE) is used.
  2. Enter a Calendar ID to specify the time periods for which rush hour data should be computed. Note that setting up a calendar with more than 7 days in cycle does not make sense for this process.
  3. The Source Region Group ID, Source Location Profile ID, Destination Region Group ID, and Destination Location Profile ID determine the region geographies for which rush hour data is generated. Depending on which of these fields have data specified, the process evaluates ESE rush hour data as explained above. If some of these lanes already exist, they will be identified and used. Otherwise, new lanes will be generated by the process.
  4. The Consider Lanes Bi-directional checkbox indicates that the process considers all lanes as specified and then reverses them. For example, a lane from Source A to Destination B (A > B) is used and then reversed as B > A and both are used to calculate rush hour.

    This defaults to selected.

    Selecting this option doubles the runtime for the Generate Rush Hour Data process, as the process evaluates twice the amount of lanes.

    This option is ignored if only Source Region Group and/or Source Location Profile are specified.
  5. The Scenario ID is used as an identifier by which the generated rush hour data can be identified and controlled.
  6. The Rush Hour Delay Threshold determines the duration input. This specifies the minimum Rush Hour Delay time needed to create an entry in the RUSH_HOUR table. For example, for a difference of 5 minutes, you may not care.
  7. Enter a User Role. When you run, publish, or schedule a recurring process, you can specify the user role used to perform the process, just like when you run agents. The user role can be used to specify a separate VPD role or another domain to run the agent. When a recurring process is modified, the user role of the recurring process can be modified as well.
  8. Select a Log ProfileÂ to generate focused logging when reproducing an error for the specific action. You cannot select a log profile when you choose to schedule a process, as automatic generation of action logs is not supported.
  9. Select Execute, Publish, or Schedule to determine when the process will be executed.
  10. Click Submit.
