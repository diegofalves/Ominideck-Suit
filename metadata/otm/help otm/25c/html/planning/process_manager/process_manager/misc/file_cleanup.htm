File Cleanup

This page is accessed via Configuration and Administration > Process Management > Configuration and Administration > File Cleanup.

Note: You must be logged in as a DBA to access this page.

Use this process to clean up managed files. When you run it, the systemÂ deletes any managed file matching the file lifecycle, whose age exceeds the Older Than field.

Note:Â A reserved recurring processÂ is staged for file cleanup with no File Lifecycle or Older Than specified, running hourly. This effectively removes both temporary files older than their maximum lifespan and runs through usage checks for (total) usage. This is important as usages may be exceeded after files are added to management, as they grow.

As user and workflow actions add files within Oracle Transaction Management, File Management monitors resource use. When resources are exceeded, the system automatically cleans up old files to provide resources for new ones. Three types of resource usage are checked: Total disk space, By Use Case file count and an Overall Performance file count.

Total resources are checked to avoid excessive disk space use, defaulting to a 50 GB maximum used by Oracle Transportation Management tracked files. When exceeded, the system runs each of the following steps until disk space is under the maximum.

  1. Delete all temporary files older than their minimum lifespan.
  2. Delete alert-managed files older than their maximum lifespan, one by one, until sufficient space is reclaimed.
  3. Delete temporary files larger than 100 KB, one by one, until sufficient space is reclaimed. Files are deleted oldest to newest.
  4. Delete alert-managed files larger than 100 KB, one by one, until sufficient space is reclaimed. Files are deleted oldest to newest.
  5. Delete all temporary files, one by one, until sufficient space is reclaimed. Files are deleted oldest to newest.
  6. Delete all alert-managed files, one by one, until sufficient space is reclaimed. Files are deleted oldest to newest.

These steps ensure that disk space maximums are almost never exceeded. Â 

Use Case resources are checked to avoid excessive file creation over a short period. Each file management use case maintains a file count maximum, defaulting to 1000 files per day over the use case's maximum lifespan. E.g., if a use case has a maximum lifespan of 30 D, it is allocated a maximum file count of 30,000 files. The goal is to avoid accidental short-term creation of tens of thousands of files. On new file creation for a particular use case, the system runs each of the following steps until the file count is under the maximum:

  1. Delete all use case files older than their maximum lifespan.
  2. Delete all use case files older than their minimum lifespan.

These steps ensure use case file maximums only block new file creation when too many files for that use case are created within the minimum lifespan. Note that when a use case is blocked by file usage, it has no effect on any other use case. File creation for other activity is not blocked.

Total file count is checked to try to limit the overall number of files to 250,000. If exceeded, the system runs each of the following steps until the total file count is under 250K:

  1. Delete all temporary files older than their maximum lifespan.
  2. Delete all temporary files older than their minimum lifespan.
  3. Delete alert-managed files older than their maximum lifespan, one by one, until sufficient file count is reclaimed.

If the total file count remains over the performance limit, file creation is not blocked. System performance may be affected, but the system will continue to add new files. Â 

While these algorithms maximize resource availability for new files, they only apply to temporary and alert-managed use cases. They also are triggered only when files are added. If file growth in a single file causes excessive resource use, the file addition algorithm may lag in revering resources. The File Cleanup workflow addresses both of these issues.

When you run the File Cleanup process without specifying an Older Than value, total disk space is checked. If exceeded, the system runs each of the following steps until disk space is under the maximum:

  1. Delete all temporary files older than their maximum lifespan, if temporary files are included in the File Lifecycle selection or the selection is left empty.
  2. Delete all temporary files older than their minimum lifespan, if temporary files are included in the File Lifecycle selection or the selection is left empty.
  3. Delete alert-managed files older than their minimum lifespan, one by one, if alert-managed files are included in the File Lifecycle selection or the selection is left empty.

Cleaning Up Files

  1. Select the File Lifecycle. The options include:
    • Temporary: Files that are automatically removed when they exceed their maximum lifespan. These will be automatically removed when exceeding minimum lifespan if space is needed.
    • Alert-Managed: Files that are automatically removed when space is needed. Warnings are logged as space constraints for these files are approached.
    • Caller-Owned: Files that are not automatically removed but can be deleted via the File Cleanup process.
  2. Enter the required number of days (D), hours (H), and minutes (M) in the Older Than section. Records older than the mentioned duration will be considered for removal.
  3. Enter a User Role. When you run, publish, or schedule a recurring process, you can specify the user role used to perform the process, just like when you run agents. The user role can be used to specify a separate VPD role or another domain to run the agent. When a recurring process is modified, the user role of the recurring process can be modified as well.
  4. Select a Log ProfileÂ to generate focused logging when reproducing an error for the specific action. You cannot select a log profile when you choose to schedule a process, as automatic generation of action logs is not supported.
  5. Select Execute, Publish, or Schedule to determine when the process will be executed.
  6. Click Submit.
